# Сравнительный отчет: n8n (No-Code) vs LangChain (Code-First)

## 1. Эволюция подхода: от Python к n8n
Ранее в **Проектах 12 и 17** я уже работала с погодными данными, используя Python и библиотеки для работы с API. 
* **Сравнение:** В коде (Проект 17) мне приходилось вручную прописывать логику связки двух API-запросов (координаты + погода) и обрабатывать JSON-ответы через циклы и словари. 
* **В n8n:** В n8n я построила четкий сценарий: поиск координат (Request1) -> запрос погоды (Request2), что оказалось гораздо быстрее и понятнее, при этом сохранилась та же точность данных.

## 2. Работа с данными и API
* В **n8n** работа с JSON реализована через выражения (Expressions). Настройка интеграции с Open-Meteo в n8n заняла в 3 раза меньше времени за счет визуального конструктора заголовков и параметров. Это удобно для простых преобразований, но становится громоздким при сложной математической обработке.
* В **LangChain** (Python) манипуляция данными через функции дает больше гибкости, особенно при работе с большими массивами информации или кастомными фильтрами. Но иногда приводит к галлюцинациям в параметрах запроса.

## 3. Обработка ошибок (Error Handling)
Реализация `Error Trigger` в n8n — не нужно проверять каждую деталь по отдельности. Мне удалось настроить оповещение о сбое во всей системе буквально одной нодой. В коде для этого требуется прописывать блоки `try-except` для каждого шага интеграции.

## 4. Выводы
* **n8n** — идеальный инструмент для быстрой автоматизации бизнес-процессов, создания прототипов и простых уведомлений (как погодный бот).
* **Code-First подход** остается незаменимым для создания сложных RAG-систем, многоагентных сред с глубокой логикой и высоконагруженных приложений.

Для текущей задачи (Weather Agent) n8n оказался более эффективным решением благодаря скорости развертывания и встроенным интеграциям.
